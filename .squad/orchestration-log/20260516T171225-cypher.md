### Azure Functions Stable Port — Root Cause & Resolution

**Problem:** Azure Functions resource in Aspire assigned random proxy port on every run, preventing stable local endpoint prediction.

**Investigation:** Initial approach was to add `Properties/launchSettings.json` with fixed `applicationUrl`. Testing revealed this does NOT work for Azure Functions isolated worker model—the launch settings are ignored by the Functions host runtime when orchestrated by Aspire.

**Resolution:**
1. Delete `src/JosephGuadagno.Broadcasting.Functions/Properties/launchSettings.json`
2. Delete empty `Properties/` directory
3. Add `.WithHttpEndpoint(port: 7071, isProxied: false)` to Functions resource in `AppHost.cs`
   - `isProxied: false` is the correct Aspire pattern for Azure Functions
   - Functions host binds directly to 7071, bypassing Aspire's reverse proxy layer
   - Stable across all local runs

**Verification:** `dotnet build .\src\ --no-restore --configuration Release` — Build succeeded, 0 errors.
